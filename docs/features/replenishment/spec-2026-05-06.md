## 1. Проблема и цель

### Проблема
Сегодня заказ сети E-Zoo формируется в значительной мере вручную: категорийный менеджер вычитывает остатки и заказы в пути из ERP, на глаз корректирует прогноз, согласует лимиты с менеджером закупок и собирает черновик заказа в X-Flow / ERP. Это даёт три устойчивых дефекта:

1. **Out-of-stock (OOS)** на популярных SKU — особенно в окнах между поставками и при сезонных всплесках, которые менеджер не успевает отработать.
2. **Overstock и зависание капитала** на хвосте ассортимента — лимиты применяются непоследовательно, страховые запасы накручиваются «на всякий случай».
3. **Низкая объяснимость** решений: после факта невозможно ответить, почему данный SKU не был заказан или почему был заказан в избыточном объёме.

### Цель
Replenishment-система автоматически формирует **черновик плана заказа** для пары `(supplier, scheduled_slot)` в виде структурированного объекта `replenishment_plan` и публикует его в X-Flow / ERP по REST API с вердиктом `auto` либо `manual_review`. Ответственность за финальную сборку EDI-документа `ORDERS` остаётся за X-Flow (фаза 3) — Replenishment отвечает только за решение «что, сколько, у кого, когда».

### Метрики успеха
Дифференцированные SLA по ABC-классам, горизонт достижения — 6 месяцев после промышленного запуска:

| KPI | A-класс | B-класс | C-класс |
|---|---|---|---|
| **OSA** (On-Shelf Availability) | ≥ 98% | ≥ 95% | ≥ 90% |
| **Stock days** | ≤ 30 | ≤ 45 | ≤ 60 |
| **OTIF** (On-Time In-Full) | ≥ 90% | ≥ 90% | ≥ 90% |
| **Доля автозаказов** (`was_auto_ordered = true` без `was_manual_override`) | ≥ 80% | ≥ 80% | ≥ 80% |

Все четыре метрики берутся из витрины `mart_kpi_daily` (см. контракт витрин §3.3) и считаются скользящим окном 28 дней.

### Когда фича НЕ нужна (исключения из скоупа)
| Кейс | Поведение |
|---|---|
| Новый SKU без истории (< 28 дней продаж) | Замолкаем, рассчитываем только если кат. менеджер явно прописал SKU-аналог в `product_features.analog_sku`; иначе ручной режим |
| Распродажа / `lifecycle_state = phase_out` | Не заказываем вовсе; `failed_lines` не возникает |
| Интрадей-расчёты (на каждый чек / каждое движение) | Только batch-режим на MVP. Реактивная модель отклонена в Блоке 2 |
| Магазины / склады в статусах `pilot` или `closed` | Полностью изолированы из расчёта; включаются вручную при переводе в `active` |

## 2. Пользователи

Опираемся на участников из README_summary §2 и пользовательскую матрицу PRD «Адаптер источников» §3.

| Роль | Что делает в Replenishment | RBAC-класс |
|---|---|---|
| **Категорийный менеджер E-Zoo** | Главный бизнес-владелец. Видит план в X-Flow/ERP UI, правит вручную (`manual_overrides`), запускает форс-расчёт по своим категориям, наблюдает KPI в Grafana | `category_manager` |
| **Менеджер закупок E-Zoo** | Согласует лимиты `order_rule` (sum/pallets/weight/volume), редактирует `supplier_specs.priority`, исключения для VIP-поставщиков | `category_manager` (на MVP) или `admin` для редактирования rules |
| **Оператор закупок (smoke)** | Ручной режим для новых SKU и аномалий | `category_manager` |
| **DevOps / SRE (наша команда)** | Грузопроцессы, healthcheck, alert-routing, ручной retry упавших runs | `admin` |
| **IT-отдел E-Zoo** | Read-only KPI и runs-history, эскалации | `viewer` |
| **X-Flow ETL / Replenishment Worker** | Машинный consumer API: POST /runs, GET /runs/{id}, GET /plans/{run_id} | сервисный JWT (m2m) |

> **Различия в поведении:**
> - `viewer` видит KPI и список runs, но **не видит** закупочные цены и индивидуальные скидки поставщиков (column-level mask).
> - `category_manager` ограничен своим `scope` (через `order_rule.scope.category_paths`): чужая категория → 404 на чтение, 403 на write.
> - `admin` редактирует `order_rule` и сервисные настройки (расписание, threshold устаревания витрин и т. п.).

## 3. Сценарии использования

### 3.1. Happy path: плановый расчёт по расписанию поставщика

1. Cron внутри Replenishment поднимается каждые 5 минут и проверяет таблицу `supplier_calendar(supplier_id, weekday, slot_time, cutoff_at)`.
2. Когда наступает slot для поставщика `S`:
   - Сервис формирует `run_id = uuid()`, пишет `runs(id, started_at, supplier_id=S, trigger_type='schedule', triggered_by='system', status='queued', rules_load_id=<snapshot>, marts_load_id=<snapshot>)`.
   - Загружает срезы витрин `mart_demand_daily`, `mart_inventory_state`, `mart_orders_in_transit`, `mart_supplier_specs` (из контракта §3) для актуального `marts_load_id`.
   - Параллельно для каждой пары `(location, sku)` в скоупе поставщика: вычисляет прогноз → применяет `order_rule` → проверяет лимиты.
   - Собирает `replenishment_plan` и сохраняет в `replenishment_plans(run_id, supplier_id, lines, summary, status, decided_verdict)`.
3. Финал: статус `runs.status = 'committed'`, плановый объект публикуется через REST: X-Flow / ERP опрашивает `GET /v1/runs/{run_id}` (polling) или забирает план через `GET /v1/plans/{run_id}`.
4. Категорийный менеджер видит план в UI X-Flow/ERP, правит при необходимости (правки уходят в `manual_overrides`, флаги `was_manual_override` обновляются в `mart_kpi_daily` после фазы 4).
5. До `cutoff_at` X-Flow собирает EDI-`ORDERS` из плана и отправляет поставщику.

### 3.2. Edge case: авто-форс по падению OSA

Триггер: «внеплановый расчёт» из swimlane-диаграммы фазы 2. Реализация:

- Раз в 30 минут фоновый job сравнивает текущий `mart_kpi_daily.osa_pct` с целевым по ABC-классу.
- Если `osa_pct ≤ target − 5 пунктов` (например, A-SKU имеет цель 98%, fact 92% → trigger), сервис отбирает все пары `(location, sku)` ниже порога.
- Группирует их по поставщику. Для каждого поставщика, у которого **сегодня есть свободный слот в `supplier_calendar`** и у которого нет уже-running run-а (см. §3.5 идемпотентность), запускает внеплановый run с `trigger_type='osa_drop'`.
- Если ближайший слот уже прошёл — пишет `audit_event` и пропускает; кат. менеджер видит уведомление в Grafana.

### 3.3. Edge case: ручной форс категорийным менеджером

Кат. менеджер (через X-Flow / ERP UI или admin-CLI) дёргает `POST /v1/runs` с телом `{supplier_id, scope: {locations?, skus?}, trigger_type: 'manual', reason: 'free-text'}`. Бэкенд проверяет `category_paths` менеджера ⊇ переданный `scope`, иначе 403.

### 3.4. Прерванный сценарий (atomic restart)

Рантайм или БД упали в середине расчёта:
- `runs.status` переводится в `failed` (через recovery-job на старте сервиса: ищет `runs.status='running'` старше 30 минут).
- Никаких частичных плановых объектов наружу — `replenishment_plans` для этого `run_id` либо отсутствует, либо имеет `status='discarded'`.
- Auto-retry по экспоненциальному бэкоффу (см. §6.3): через 5с × {1, 2, 4, 8, 16}, max 5 попыток. После исчерпания — `runs.status='failed_permanent'`, алерт в SRE и кат. менеджеру.
- Ручной retry от admin: `POST /v1/runs/{failed_run_id}/retry` создаёт новый `run_id`, наследует `supplier_id` и `scope`, но берёт **актуальный** `marts_load_id` (не воспроизводит исторический срез).

### 3.5. Идемпотентность

`POST /v1/runs` принимает заголовок `Idempotency-Key`. При совпадении ключа за окно 24 часа сервис возвращает существующий `run_id` вместо создания нового. Без заголовка ключ автоматически вычисляется как `hash(supplier_id + scheduled_slot + trigger_type)`, что гарантирует идемпотентность плановых runs из cron.

## 4. Технические ограничения

### 4.1. Объёмы и SLO
- **Расчётный вектор (ориентир):** ~150 локаций × ~30 000 SKU × горизонт 14 дней.
- **SLO на один run на одного поставщика:** ≤ 15 минут (p95) от `runs.started_at` до `runs.committed_at`.
- **SLO API (синхронные):** `POST /v1/runs` → 200 OK с `run_id` за ≤ 200 мс (создание записи + queue, без блокировки на расчёт).
- **Concurrency:** ≥ 8 параллельных runs (по разным `supplier_id`) на инстанс worker-а.

### 4.2. Синхронность
- Расчёт **асинхронный** (batch). API `POST /v1/runs` ставит задачу в очередь (PostgreSQL `SELECT ... FOR UPDATE SKIP LOCKED`).
- Polling-модель: клиент опрашивает `GET /v1/runs/{id}` для статуса. Webhook-модель отклонена на MVP (избыточно — X-Flow и ERP уже встроены в общий poll-цикл).
- Server-Sent Events / WebSocket для прогресса отклонены (избыточно для batch-системы; см. §10).

### 4.3. Консистентность vs скорость
**Приоритет — консистентность.** Replenishment не публикует план «по чуть-чуть»: либо все строки рассчитаны для всех пар скоупа, либо `runs.status='failed'`. Per-SKU isolation допускает помеченные `failed_lines` (см. §6), но требует, чтобы доля битых строк была ≤ 5% от общего объёма скоупа — иначе hard-fail всего run-а.

### 4.4. Хранение и версионирование прогнозов
Таблица `forecasts(run_id UUID, location_id, product_id, forecast_date DATE, expected_qty NUMERIC, p10 NUMERIC, p50 NUMERIC, p90 NUMERIC, model_version TEXT, PRIMARY KEY(run_id, location_id, product_id, forecast_date))`. Версия прогноза = `run_id`. Каждый run пишет полный новый срез — **overwrite не делается**. Калибровка модели (MAPE/bias) ретроспективна: сравниваем прошлые `forecasts.expected_qty` с пришедшим позже `mart_demand_daily.qty_sold`.

Bi-temporal (как у фактов в Адаптере) для прогнозов **не делаем** — overhead не оправдан, прошлые runs хранятся «как есть» и партиционируются по `run_id` помесячно.

### 4.5. Внешние зависимости
**Только PostgreSQL 18.** Никакого Redis, RabbitMQ, S3, Kafka на MVP:
- Очередь runs — таблица `runs_queue` + `SKIP LOCKED`.
- Аудит-лог — таблица `audit_events`.
- Хранение планов — таблица `replenishment_plans`.
- Прогнозы — таблица `forecasts`.

Внешние **источники данных:**
- Витрины X-Flow ETL (`mart_*`) — read-only, в той же БД (схема `marts`) либо через FDW/реплику. Контракт схемы — в `data-marts/contract-2026-05-06.md`.
- Никаких прямых вызовов в ERP / Адаптер источников / EDI-шлюз.

### 4.6. Поведение при устаревании витрин
Перед стартом расчёта worker проверяет `mart_load_metadata.last_acquired_at` для каждого требуемого мартa. Если самый старый `last_acquired_at < now() - 24h`:
- `runs.status = 'failed_marts_stale'`,
- supportMessage `REPL-MARTS-STALE`,
- алерт в SRE и кат. менеджеру (Grafana → Alertmanager),
- никаких частичных расчётов.

Threshold 24ч — параметр `config.replenishment.marts_stale_threshold` (по умолчанию `24h`, override per-supplier через `supplier_specs.marts_freshness_sla`).

### 4.7. Инфраструктурные изменения
- Новый Go-сервис `cmd/replenishment` на стандартном стеке проекта (Go 1.26 + Fiber v3 + pgx/v5 + golang-migrate). Структура — по образцу будущих фич: `internal/features/replenishment/{handler,service,repository,models,router,sqls}`.
- Миграции в `migrations/`: `forecasts`, `runs`, `runs_queue`, `replenishment_plans`, `audit_events`, `manual_overrides`. Ключевое использование PG18: партиционирование `forecasts` и `replenishment_plans` по `run_id` с virtual generated `run_month`.
- Prometheus-метрики (`/metrics`): runs total/failed/duration, lines_calculated total, marts_age_seconds, forecast_mape (gauge per ABC).
- Новые env vars: `REPLENISHMENT_MARTS_DSN`, `REPLENISHMENT_WORKER_CONCURRENCY`, `REPLENISHMENT_RUN_TIMEOUT`, `REPLENISHMENT_FORECAST_HORIZON_DAYS`.
- CI: добавляются интеграционные тесты на dockertest/v3 с мок-витринами.

## 5. Безопасность и доступ

### 5.1. RBAC
Три роли в `users.role`:

| Роль | Чтение | Запись |
|---|---|---|
| `viewer` | KPI-витрины, список runs (без cost-полей), агрегаты планов | — |
| `category_manager` | Всё своё (`scope ⊆ user.allowed_category_paths`), включая cost | Форс-запуск runs, `manual_overrides` своих категорий |
| `admin` | Всё | Редактирование `order_rule`, `supplier_calendar`, ручной retry любых runs, разблокировка пилотных локаций |

Авторизация — JWT (как в проекте: middleware `auth.RequireRole`). Сервисный m2m токен для X-Flow приравнивается к `admin` по эффективным правам, но логируется отдельным `subject_type='service'` в `audit_events`.

### 5.2. Чувствительные данные
**Закупочные цены** (`supplier_specs.cost_per_unit`) и **индивидуальные скидки** (`supplier_specs.discount_terms`) маскируются:
- Column-level: PG VIEW `replenishment_plans_safe_v` без cost-полей; контроллер выбирает view по роли.
- На MVP — единая стратегия маскирования; row-level security через PG18 RLS откладываем (см. Q-001).

### 5.3. Аудит
Таблица `audit_events(id, occurred_at, run_id?, subject_id, subject_type, action, before JSONB, after JSONB, reason TEXT)`:
- **Каждый run:** `action='run_started'` / `'run_committed'` / `'run_failed'` с метаданными (rules_load_id, marts_load_id, trigger_type).
- **Каждая ручная правка:** `action='line_overridden'` с before/after значений qty / supplier_id и обязательным `reason` (free-text).
- **Изменения `order_rule` (admin):** `action='rule_updated'` / `'rule_created'` / `'rule_deactivated'` с diff scope/formula/limits.
- **Чтения** в audit **не пишем** на MVP (избыточно).

Таблица иммутабельна (`INSERT ONLY`, без UPDATE/DELETE; чистка ретеншн-задачей по `occurred_at < now() - 18 months`).

### 5.4. Несанкционированный доступ
- **404** при чтении объекта вне `scope` пользователя — не раскрываем сам факт существования (`GET /v1/plans/{run_id}` для чужого supplier-а → 404).
- **403** при попытке write-операции — пользователь должен видеть, что у него «прав не хватает», а не «объекта нет» (`POST /v1/runs` с supplier-ом вне scope → 403 + supportMessage).
- Сервисные m2m токены не получают разрешение «на всё»: их `allowed_supplier_ids` фиксируется в самом JWT-claims.

## 6. UI/UX требования

### 6.1. Поверхности
**На MVP:**
- **Replenishment экспортирует REST API** (`/v1/runs`, `/v1/plans/{run_id}`, `/v1/forecasts/{run_id}`, `/v1/order-rules`) и admin-CLI (`replctl`).
- **Просмотр и правка плана** — в существующем UI X-Flow / ERP. Replenishment не строит собственный Web UI на MVP.
- **KPI-дашборд для кат. менеджера** — Grafana (новые панели в проектном дашборде E-Zoo): OSA по ABC, Stock days, OTIF, доля автозаказов, MAPE прогноза, runs/min, marts_freshness.

**Web UI Replenishment** (Vue) — отложен на v2 (см. §10), пока правка делается в X-Flow / ERP.

### 6.2. Статус долгого run-а
- `POST /v1/runs` → `200 OK`, тело `{run_id, status_url, eta_seconds}`.
- `GET /v1/runs/{run_id}` → `{run_id, supplier_id, status: 'queued'|'running'|'committed'|'failed'|'failed_permanent'|'partial', started_at, finished_at?, lines_total, lines_failed, error?: {code, message}}`.
- Polling рекомендованный интервал — **5 секунд**, hard-cap клиентского poll — каждые 2 секунды (rate limiter в middleware).
- ETA вычисляется как `avg_run_duration_28d × scope_size_factor`; для нового supplier-а возвращается дефолт 600 секунд.

### 6.3. Идемпотентность
- Заголовок `Idempotency-Key: <uuid|user-string>` (≤ 128 байт, RFC4122 либо ASCII-printable).
- При совпадении ключа в окне 24ч — `200 OK` с тем же `run_id` (даже если старый run уже finished). Тело отличается флагом `idempotent_replay: true`.
- Без заголовка — auto-key `hash(supplier_id, scheduled_slot_iso, trigger_type)` для cron-runs; ручные runs (`trigger_type='manual'`) без явного ключа всегда создают новый run_id.
- Конфликт по дублю с разным телом запроса (другой scope) → `409 Conflict` + supportMessage `REPL-IDEMPOTENCY-CONFLICT`.

## 7. Обработка ошибок

### 7.1. Гранулярность
**Per-SKU isolation:**
- Битые строки (`forecast_failed`, `rule_missing`, `data_inconsistent`) идут в `replenishment_plans.failed_lines[]` со структурой `{location_id, product_id, error_code, message}`.
- Остальные строки публикуются нормально, `replenishment_plans.status='partial'` и `runs.status='committed_partial'`.
- **Hard threshold:** если `lines_failed / lines_total > 5%`, run полностью failed (`status='failed_quality'`), план не публикуется, supportMessage `REPL-QUALITY-LOW`.
- Per-supplier isolation реализуется естественно: на MVP один run = один supplier.

### 7.2. Коды ошибок
Все коды наследуют формат `pkg/errorspkg.ErrorResponseJSON` (см. PRD «Адаптер источников» §10) — поля `code`, `message`, `supportMessage`, `traceId`.

| Код | HTTP | Когда | Что видит пользователь | Действие системы |
|---|---|---|---|---|
| `REPL-MARTS-STALE` | 503 на `POST /v1/runs`, `failed` в статусе run-а | `mart_load_metadata.last_acquired_at < now() - 24h` для одной из требуемых витрин | «Витрины устарели, попробуйте позже или дождитесь обновления» + ссылка на dashboard freshness | run не стартует / прерывается; алерт в SRE |
| `REPL-RULE-MISSING` | строка в `failed_lines` | Для пары `(location, sku)` нет ни одного matching `order_rule` | В UI X-Flow строка помечена «правило не найдено» | SKU попадает в `failed_lines`, остальной план собирается |
| `REPL-LIMIT-EXCEEDED` | флаг в `replenishment_plans.summary.warnings` | После cap по `order_rule.limits` обрезано > 30% от расчётного `expected_qty` | Желтый бейдж «Лимит ужал заказ на N%» в UI | Вердикт `manual_review` обязателен; алерт в Grafana |
| `REPL-FORECAST-LOWCONF` | флаг в строке плана | `(p90 - p10) / p50 > 1.5` или `historical_data_days < 28` | Серый ярлык «Прогноз неуверенный» в UI | Вердикт `manual_review` для строки; в KPI идёт без штрафа |
| `REPL-IDEMPOTENCY-CONFLICT` | 409 | Тот же `Idempotency-Key`, но различный body | Стандартное Conflict-сообщение + traceId | Запись в audit |
| `REPL-FORBIDDEN` | 403 | Write вне scope | «Недостаточно прав для категории X» | Audit |
| `REPL-NOT-FOUND` | 404 | Read вне scope (или объекта правда нет) | «Объект не найден» | Без логирования цели запроса |
| `REPL-QUALITY-LOW` | run failed | `lines_failed / lines_total > 5%` | «Слишком много битых строк, план не опубликован» | run failed, без partial-публикации |

### 7.3. Retry-политика
- **Внутренний retry** (внутри одного run-а на временные failures: pgx ECONNRESET, deadlock, lock timeout) — экспоненциальный backoff `5s × {1, 2, 4, 8, 16}`, max 5 попыток на одно место. После исчерпания — run failed.
- **Run-level retry** (после `runs.status='failed'`):
  - **Auto-retry** только для `failed`-причин из allow-листа (`marts_stale`, `pg_connection_lost`, `worker_oom`). Backoff: `60s, 300s, 1500s` (1, 5, 25 минут), max 3 попытки. После — `failed_permanent`.
  - **Ручной retry** через `POST /v1/runs/{id}/retry` (admin). Создаёт новый run_id с тем же supplier/scope, но свежий marts_load_id.
- Resume-by-checkpoint **не поддерживается** (отклонено в Блоке 2 — конфликтует с atomic restart).

## 8. Принятые компромиссы

### 8.1. Модель прогноза (MVP)
Сознательно простая, без ML:

| Подмодуль | MVP-решение |
|---|---|
| **Регулярные SKU** | `expected_qty[d] = SMA28(daily_demand[d-28..d-1]) × seasonality_week_factor[weekday(d)]` |
| **Сезонные SKU** | Тот же SMA28, но с year-over-year коэффициентом из той же недели прошлого года |
| **Промо** | Поверх базы — `promo_uplift_factor` из таблицы `product_features.promo_factor` (заполняется кат. менеджером вручную или из ERP) |
| **Доверительный интервал** | `p10/p50/p90` через эмпирические квантили остатков SMA28 за последние 90 дней |
| **Новые SKU (< 28 дней)** | Не прогнозируем — `failed_lines` с code `REPL-FORECAST-NEW-SKU`; ручной режим через указание `analog_sku` в `product_features` |

Логика реализуется в Go в `internal/features/replenishment/forecasting/`. Никакого Python / LightGBM / Prophet на MVP.

### 8.2. Калькулятор заказа (MVP)
Поддерживаются только три формулы из `order_rule.formula` (см. PRD «Адаптер источников» §5.1.9):
- `reorder_point` — `target_qty = max(0, (forecast_horizon × daily_demand + safety_stock) − on_hand − in_transit)`
- `min_max` — если `on_hand + in_transit < min_qty` → заказываем до `max_qty − on_hand − in_transit`
- `econ_order_qty` (EOQ-классика) — реализация по Уилсону без оптимизатора (формула в коде, без LP-солвера)

DSL-парсер не реализуется — формулы перечислены в Go-коде в `internal/features/replenishment/calculator/formulas.go`.

### 8.3. Лимиты (cap)
**Greedy by ABC:** при превышении `order_rule.limits` сначала обрезаем C-класс, потом B, потом A. Внутри класса — по возрастанию `expected_qty / max_qty` (наименее «нужные» строки урезаются первыми). Не оптимум — приемлемая аппроксимация для MVP.

### 8.4. Выбор поставщика
**Sort by `supplier_specs.priority` ASC**, при равенстве priority — по минимальному `cost_per_unit` (если поле доступно). Сложного cost-aware выбора (с учётом доставки, holding cost, обмена) **не делаем** на MVP.

### 8.5. Технический долг (зафиксированный)
| Долг | Когда возвращаемся |
|---|---|
| Greedy-cap вместо knapsack/LP | После запуска, если `REPL-LIMIT-EXCEEDED` срабатывает >5% runs |
| Priority-only выбор поставщика | После калибровки KPI и появления cost-data |
| Нет backfill / replay прошлых runs при изменении `order_rule` | Принципиально — обновлённое правило применяется со следующего расчёта |

## 9. Отклонённые решения

| Решение | Почему отклонено |
|---|---|
| Реактивный расчёт на событие `rep.stock.changed` (на каждое движение остатков) | Лавина дублирующих runs на пиковом трафике, риск гонок, для MVP не добавляет ценности — авто-форс по падению OSA закрывает критичные кейсы |
| Resume-by-checkpoint при падении run-а | Конфликтует с atomic restart, требует прогресс-трекера, усложняет код. Auto-retry свежим run-ом проще и безопаснее |
| Webhook-нотификации в X-Flow / ERP о готовности run-а | Требует регистрации webhook-эндпоинтов на стороне X-Flow / ERP + retry-логики; polling уже встроен в их цикл |
| Server-Sent Events / WebSocket для прогресса run-а | Избыточно для batch-системы с SLO ≤ 15 минут; `GET /v1/runs/{id}` каждые 5 секунд достаточно |
| Best-effort partial publication (опубликовать что есть с флагом `partial`) при OOM/DB lost | Риск, что X-Flow сложит EDI ORDERS из неполного плана и пропустит критичные SKU |
| Bi-temporal (`valid_time + system_time`) для `forecasts` | Overhead не оправдан; снапшот-версионирование по `run_id` достаточно |
| ML-прогноз (LightGBM / Prophet / нейросети) на MVP | Другой стек (Python), сложнее отладка, риск «черного ящика» при первых внедрениях. Можем вернуться после калибровки SMA28 |
| Self-service Web UI для редактирования `order_rule` | Редактирование пока через ERP/X-Flow или admin-API; UI стоит дороже бизнес-ценности на MVP |
| Multi-echelon (склад → магазины) в одном расчёте | Существенно усложняет калькулятор; на MVP работаем «плоско» по парам `(location, sku)` |
| Single-role auth (только `user` / `admin`) | Не даёт сегрегации «своя категория», виден закупочный cost всем — нарушение требования §5.2 |
| Auth полностью внешним IAM (без своих ролей) | Разрыв с проектным паттерном (`auth.RequireRole` middleware), отложим до общей IAM-стратегии |

## 10. Открытые вопросы

> Каждый `Q-NNN` будет закрыт одноимённым `ADR-NNN` на стадии Design.

1. **Q-001.** Маскирование закупочных цен и скидок поставщиков для роли `viewer` — column-level через VIEW-ы или PostgreSQL 18 Row-Level Security? Контекст: PG18 поддерживает RLS «из коробки», но добавляет расходы на `pg_get_role_oid` в каждом запросе. Нужна оценка стоимости в QPS-нагрузке Grafana-дашборда. Дизайн-агент должен сравнить два подхода и зафиксировать.

2. **Q-002.** Связь с `OQ-8` PRD «Адаптер источников» (маркировка ветпрепаратов): должен ли калькулятор учитывать `requires_marking = true` при выборе поставщика или при формировании партии? Если да — нужно поле в `supplier_specs` (умеет ли поставщик отгружать с маркировкой) и фильтр в шаге выбора поставщика. Решение зависит от ответа юриста / QA E-Zoo по `OQ-8`.

3. **Q-003.** Каннибализация / substitutes на OOS (`OQ-9` PRD «Адаптер источников»): должен ли прогноз умножать спрос на коэффициент переключения, когда товар-аналог в OOS? На MVP мы прогноз НЕ корректируем — `lost_sales_estimate` в `mart_kpi_daily` считается «как есть». Дизайн-агент должен подтвердить, что для `lost_sales` используется чистый прогноз без compensation, и зафиксировать это в ADR (с пометкой «решение пересмотреть после v2»).

4. **Q-004.** Стратегия миграции `forecasts` и `replenishment_plans` при росте: партиционирование по `run_month` (PG18 declarative partitioning) предполагает retention-политику. Какой retention для `forecasts` (предлагаем 18 месяцев — для калибровки year-over-year), для `replenishment_plans` (предлагаем 36 месяцев — аудит и compliance), для `audit_events` (предлагаем 18 месяцев)? Подтвердить с юристом E-Zoo.

5. **Q-005.** Формат `Idempotency-Key` для случая `trigger_type='osa_drop'`: автоматически вычисляемый ключ должен исключать дубли на коротком горизонте (несколько ча сов), но не блокировать ручной форс позже. Предлагаем `hash(supplier_id, date(), 'osa_drop')` (на сутки). Дизайн-агент должен сверить с частотой повторных алертов OSA и предложить окно (день / 4 часа / другое).

6. **Q-006.** Mapping `order_rule.scope.category_paths` ↔ `users.allowed_category_paths`: как разрешать конфликт, если одно правило покрывает несколько категорий, а пользователь имеет права только на часть? Предлагаемый подход: пользователь видит только те `replenishment_plans.lines`, чьи `(location, sku)` попадают в его `allowed_category_paths`. Подтвердить семантику с категорийным менеджером — особенно для случаев, когда продукт принадлежит нескольким категориям (multi-category products).

7. **Q-007.** Cost-поля и валюта: `supplier_specs.cost_per_unit` — в UAH с НДС или без? README §5.1.7 говорит «Закупочная цена с НДС». Нужно явно зафиксировать в контракте витрин (потенциально расширяя `mart_supplier_specs`) и в формулах калькулятора, что `cost` включает НДС, а лимит `max_amount_uah` — тоже с НДС. Кросс-проверка с категорийным менеджером и финансами.

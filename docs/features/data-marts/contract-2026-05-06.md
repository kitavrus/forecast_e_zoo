# Контракт витрин данных (Data Marts) для Replenishment

> **Статус:** черновой контракт между тремя соседними дорожками диаграммы — **«Адаптер источников» → X-Flow ETL → Replenishment**. Описывает, какие денормализованные витрины X-Flow должен построить из наших экспортов, чтобы прогноз и калькулятор могли работать.
>
> **Это не PRD ни одной из сторон** — это пограничный артефакт. Без него:
> - Replenishment-команда не знает, на чём строить модели.
> - X-Flow ETL не знает, что нормализовать.
> Задача документа — разблокировать обе команды для параллельной работы.
>
> **Связанные документы:**
> - PRD «Адаптер источников» (вход в X-Flow): [data-export/spec-2026-05-05.md](../data-export/spec-2026-05-05.md)
> - PRD Replenishment (потребитель витрин): TBD

## 1. Контекст

**Откуда данные:** наш «Адаптер источников» отдаёт сырые таблицы через `/v1/...`:
`products`, `product_barcodes`, `category`, `location`, `store_assortment`, `supplier`, `supply_spec`, `promo`, `order_rule`, `supply_plan`, `receipt_line`, `location_stock_snapshot`, `stock_movement` (+ companion `*_details`), `supplier_stock_snapshot`, `master_change_log`, `store_assortment_lifecycle_events`.

**Куда данные:** Replenishment-модулю (прогноз + калькулятор + KPI). Прогноз ест агрегаты, калькулятор — текущее состояние, KPI — историю фактов.

**Кто строит:** X-Flow ETL (между нами).

---

## 2. Принципы дизайна витрин

1. **Денормализация против запросов.** Replenishment не должен делать 5-table JOIN на каждый прогноз — нужные поля материализуются в витрину один раз.
2. **Snapshot-модель для обучения.** Витрина для прогноза — append-only daily snapshots (`as_of_date`) — позволяет переобучать модели на любой дате без leakage из будущего.
3. **Current state для калькуляции.** Витрина для калькулятора — current snapshot (одна строка на пару `(location, product)`), обновляется ежесуточно.
4. **Source attribution.** Каждая строка витрины несёт `source_load_id` нашего адаптера + `etl_run_id` X-Flow — для воспроизводимости.
5. **Bi-temporal не пробрасываем напрямую.** Replenishment видит «как было на дату X» через `as_of_date`, не лезет в `system_time_*`. Это упрощает прогноз-код.
6. **Refresh cadence:** ежесуточно, после успешного `committed` загрузки нашего адаптера. ETL пересчитывает только текущий день + переоткрытые из-за коррекций.

## 3. Минимальный набор витрин (MVP)

### 3.1. `mart_demand_history` — обучающая выборка для прогноза
Daily агрегаты продаж per `(location, product)` за N лет.

| Поле | Тип | Источник |
|---|---|---|
| `as_of_date` | DATE | `receipt_line.event_time::date` |
| `location_id` | TEXT FK | `receipt_line.location_id` |
| `product_id` | TEXT FK | `receipt_line.product_id` |
| `qty_sold` | NUMERIC | `SUM(receipt_line.qty WHERE line_kind='sale')` |
| `qty_returned` | NUMERIC | `SUM(receipt_line.qty WHERE line_kind='return')` |
| `qty_promo_bonus` | NUMERIC | `SUM(receipt_line.qty WHERE line_kind='promo_bonus')` |
| `qty_gift` | NUMERIC | `SUM(receipt_line.qty WHERE line_kind='gift')` |
| `revenue_paid` | NUMERIC | `SUM(qty × unit_price_paid WHERE line_kind='sale')` |
| `discount_total` | NUMERIC | `SUM(discount_amount)` |
| `transactions_count` | INTEGER | `COUNT(DISTINCT receipt_id)` |
| `had_promo` | BOOLEAN | `EXISTS(promo WHERE date_from <= as_of_date <= date_to)` |
| `promo_type` | TEXT NULL | первый активный promo.type, если был |
| `was_in_assortment` | BOOLEAN | строка в `store_assortment` существовала на дату |
| `lifecycle_state_at_date` | TEXT | состояние из `store_assortment_lifecycle_events` |
| `was_oos` | BOOLEAN | `qty_on_hand=0` на конец дня |
| `source_load_id`, `etl_run_id` | UUID | provenance |

**Назначение:** обучение моделей прогноза (sales velocity, lift при промо, OOS-correction).

### 3.2. `mart_calculation_input` — current state для калькулятора
Одна строка на активную пару `(location, product)`.

| Поле | Тип | Источник |
|---|---|---|
| `location_id`, `product_id` | TEXT FK | |
| `qty_on_hand` | NUMERIC | свежий `location_stock_snapshot` |
| `qty_reserved`, `qty_in_transit` | NUMERIC | то же |
| `qty_available` | NUMERIC GENERATED | `qty_on_hand − qty_reserved + qty_in_transit` |
| `target_qty`, `min_shelf_presentation`, `facings` | от `store_assortment` | |
| `lifecycle_state` | TEXT | из `store_assortment.lifecycle_state` |
| `primary_supplier_id` | TEXT | из `supply_spec` ORDER BY priority LIMIT 1 |
| `lead_time_days`, `min_order_qty`, `purchase_price` | от primary supplier_id | |
| `next_cutoff_at` | TIMESTAMPTZ | вычислено из `supply_plan` |
| `applicable_rule_id` | TEXT | результат scope-matching `order_rule` |
| `supplier_capacity_qty` | NUMERIC NULL | `qty_available − qty_reserved_for_us` из `supplier_stock_snapshot` |
| `is_perishable`, `requires_cold_chain` | BOOLEAN | из `products` |
| `as_of_timestamp` | TIMESTAMPTZ | момент сборки витрины |

**Назначение:** калькулятор делает один SELECT на пару, без JOIN'ов.

### 3.3. `mart_kpi_daily` — KPI per `(location, product, date)`
Базовая фактологическая витрина для дашбордов.

| Поле | Источник |
|---|---|
| `as_of_date`, `location_id`, `product_id` | |
| `osa_pct` | % дня с `qty_on_hand > 0` (из intra-day снимков, если есть; иначе binary) |
| `stock_days` | `qty_on_hand / avg_daily_demand_28d` |
| `lost_sales_estimate` | OOS-периоды × прогнозный спрос |
| `was_auto_ordered` | пришёл ли калькулятор к ORDERS на эту пару |
| `was_manual_override` | был ли ручной правкой |
| `received_on_time` | OTIF из `receiving_details.qty_short` |
| `fill_rate` | `qty_received / qty_ordered` из `receiving_details` |

### 3.4. `mart_master_current` — справочники в одной плоской таблице
Денормализованный JOIN `products + category + brand + primary_supplier` для удобства аналитики и фильтров. Перестраивается раз в сутки.

### 3.5. `mart_supplier_scorecard` — supplier-level метрики
Метрики per поставщик per период (week/month):
- `fill_rate_avg`, `otif_pct`, `lead_time_actual_avg`
- `qty_short_total`, `qty_damaged_total`, `qty_returned_total`
- `lines_delivered`, `lines_late`

## 4. Refresh strategy

| Витрина | Обновление | Условие триггера |
|---|---|---|
| `mart_demand_history` | append daily | после `committed` нашего load |
| `mart_calculation_input` | full rebuild | после `committed` нашего load |
| `mart_kpi_daily` | append daily | после `mart_demand_history` |
| `mart_master_current` | full rebuild | после `committed` или change_log diff |
| `mart_supplier_scorecard` | rolling weekly + ondemand | по cron + после `receiving_details` обновлений |

**Bi-temporal коррекции:** если в нашем `master_change_log` или `receipt_line` пришла коррекция за прошлую дату — X-Flow пересчитывает соответствующие дни в `mart_demand_history` и `mart_kpi_daily` (idempotent).

---

## 5. Что НЕ в этом контракте

- Конкретный стек ETL (Airflow / dbt / dagster / кастом) — решение X-Flow команды.
- Внутренние staging-таблицы X-Flow (нас не касаются).
- Маппинг ID между источниками (когда появится multi-source) — отдельный артефакт.
- Schema-эволюция Replenishment-side моделей.

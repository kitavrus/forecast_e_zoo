# PRD + ADR: Модуль выгрузки данных (data-export adapter)

Дата: 2026-05-05
Источник: интервью `/spec-interview` (см. план `~/.claude/plans/prd-adr-sunny-cosmos.md`).

---

## 1. Context

Фича на основе [diagram_0_overview.html](../../diagram_0_overview.html), задача **«Выгрузка данных»** в дорожке ERP клиента E-Zoo (Фаза 1 — Сбор данных). На диаграмме рядом — отдельная дорожка «Адаптер источников» (1С/ERP/WMS/OMS → единый формат), и проектируем мы именно её — pull-сторону.

**Зачем сейчас:** без надёжного потока справочников и фактов из ERP E-Zoo не работают ни прогноз, ни калькуляция, ни EDI. Все KPI Replenishment (OSA, OTIF, Stock days) зависят от полноты и свежести данных на входе. Сегодня поток описан только концептуально (см. [diagram_1_data_collection.html](../../diagram_1_data_collection.html), [README_summary.md](../../README_summary.md) §5).

**Что получим:** канонический REST-сервис на Go (`/v1/...`), который ежесуточно вытягивает мастер-данные и движения из ERP E-Zoo через её REST API, материализует в PG18 и предоставляет X-Flow read-only выборки с гарантиями полноты, идемпотентности, версионируемой схемы.

**Стек:** Go 1.26 + Fiber v3 + PostgreSQL 18 + pgx/v5 + go:embed + golang-migrate (без auto-apply) + dockertest. CLAUDE.md в репо отсутствует — раздел 8 как ориентир недоступен; домен реконструирован из [README_summary.md](../../README_summary.md) и набора диаграмм.

## 2. Goals / Non-goals

**Goals (MVP):**
- Pull-адаптер для одного источника (ERP E-Zoo, REST/SOAP API).
- Полная сутечная выгрузка справочников и движений из §5 ТЗ.
- Материализация в PG18: snapshot-таблицы с `load_id`.
- REST-контракт `/v1/...` с content-negotiation (NDJSON inline / Parquet через signed URL).
- Severity-движок валидации: critical → fail load, soft → quarantine.
- Retention: 30 дней hot в PG + 365 дней cold в S3 Parquet.
- Observability: метрики Prometheus, 4 класса алертов.

**Non-goals (вне MVP, см. §11):**
- Реальный CDC (логическая репликация / outbox в ERP).
- Multi-source (WMS, OMS, DWH) — интерфейс `SourceReader` оставляем, реализация одна.
- Bi-temporal SCD-2 для **мастер-данных** (для фактов bi-temporal — да).
- Собственный ops UI / business dashboard (только Grafana и admin-API).

## 4. Functional requirements

### 4.1. Триггер
- **Гибрид:** внутренний cron (Go, в самом процессе) + admin-API `POST /admin/loads` для manual/retry.
- Окно по умолчанию — раз в сутки, время и TZ настраиваются в config.
- Lock-файл/advisory lock в PG чтобы исключить параллельный запуск.

### 4.2. Жизненный цикл прогона
1. `load` начинается → запись `loads(id, started_at, status='running', source='erp_e_zoo')`.
2. По очереди тянутся группы (см. §5): мастер-данные → факты.
3. На каждую сущность — pull через REST с курсорной пагинацией + checkpoint.
4. **Atomic restart**: при сбое весь `load_id` помечается failed, snapshot-pointer не двигается.
5. По завершении — `loads.status='committed'`, `snapshot_pointer.current_load_id := load_id`.
6. Старые `load_id` уходят на retention sweeper (sec §8).

### 4.3. Валидация (severity)
- **Critical** (load_id → failed):
  - FK violation на отсутствующий master (например, чек ссылается на неизвестную товар/баркод/store).
  - NULL в primary key.
  - Отрицательные остатки / отрицательные количества в чеке.
  - `event_time` в будущем (>now+15min, защита от часов источника).
- **Soft** (строка → `reject_log` с причиной, остальное идёт):
  - NULL в необязательном attribute (бренд, страна).
  - Невалидный формат (даты, EAN-checksum).
  - Out-of-range numeric (<0 в полях, где нет смысла, кроме остатков).
- Правила хранятся в config-файле (`validation_rules.yaml`) и загружаются на старте — без deploy.

### 4.4. PII
- На ингесте чеков `loyalty_id` → `loyalty_hash := SHA256(loyalty_id || tenant_salt)`.
- Telephone/email из чеков **не выгружаются вообще** (drop в маппинге).
- Соль хранится в Vault, ротация — не чаще раза в год (иначе теряется join на исторических данных).

### 4.5. UPSERT-семантика мастер-данных
SCD-2 для мастера отложен (ADR-003), хранится только current state. Чтобы это не превратилось в «тихую перезапись без следов» — фиксированные правила:

- **`product_id` PK** → `INSERT ... ON CONFLICT (product_id) DO UPDATE SET ...`.
- **`created_at`** — НЕ перезаписывается (`ON CONFLICT` его не трогает). Дата первого появления карточки в нашей БД.
- **`updated_at`** — перезаписывается ТОЛЬКО при реальном diff tracked-полей (через `WHERE products.* IS DISTINCT FROM EXCLUDED.*` или `BEFORE UPDATE` триггер). Не дёргается на каждом ежесуточном load.
- **`load_id`** — перезаписывается всегда, даже без diff. Нужен как маркер «строка ещё видна в источнике».
- **Все остальные поля** — перезаписываются значением из EXCLUDED (новой выгрузки).
- **Дубликат `product_id` внутри одного `load`** → **critical** (источник прислал две версии в одной выгрузке = битый экспорт).

#### 4.5.a Журнал изменений мастер-данных (`master_change_log`)
Append-only лог значимых изменений. **Это не SCD-2** — нет point-in-time восстановления; даёт только ответ «что и когда менялось». Список tracked-полей задаётся config-driven (`master_tracked_fields.yaml`), стартовый набор:
- `products`: `name`, `brand`, `manufacturer`, `weight_kg`, `pallet_qty`, `shelf_life_days`, `storage_temp_min`, `storage_temp_max`, `requires_prescription`, `is_dangerous_goods`, `category_path`, `status`
- `product_barcodes`: `product_id` (миграция баркода между карточками), `pack_qty`, `is_primary`, `country_origin`

| Поле | Тип |
|---|---|
| `event_id` | UUID PK |
| `entity` | TEXT (`products` / `product_barcodes`) |
| `entity_pk` | JSONB (например `{"product_id":"123"}` или `{"barcode":"4607..."}`) |
| `field` | TEXT |
| `old_value` | JSONB NULL |
| `new_value` | JSONB |
| `changed_at` | TIMESTAMPTZ NOT NULL |
| `load_id` | UUID FK |

Эндпоинт: `GET /v1/master_change_log?entity=...&since=...&field=...` — для X-Flow/категорийщика, чтобы видеть «что изменилось со вчера». Если позже понадобится полноценный SCD-2 — backfill из этого журнала.

### 4.6. Read-during-write
- X-Flow всегда видит последний committed snapshot (`snapshot_pointer.current_load_id`).
- Текущий запущенный `load` не виден в `/v1/...` ответах.
- Минимум 2 snapshot хранятся одновременно (текущий + предыдущий) для отката.

## 5. Data model

### 5.1. Канонические сущности (на основе [README_summary.md](../../README_summary.md) §5)

#### 5.1.1. Товары (`products`)
Базовая карточка товара (Product Master / Item Master в ERP-терминологии; «Номенклатура» в 1С). У одного товара может быть **несколько баркодов** (разные упаковки, перепакованные партии, региональные EAN). Хранятся отдельной таблицей `product_barcodes` с FK на `product_id` для нормализованного поиска и уникальности EAN.

| Поле | Тип | Notes |
|---|---|---|
| `product_id` | TEXT PK | внутренний ID E-Zoo (бывш. `sku_id`) |
| `name` | TEXT | торговое наименование |
| `brand` | TEXT NULL | Бренд (Royal Canin, Whiskas, Pro Plan…). Критично для отчётов «доля бренда в категории», фильтров категорийщика, переговоров с поставщиками, ABC-анализа по бренду. Может отличаться от производителя (бренд PetXX выпускает завод-контрактник). |
| `manufacturer` | TEXT NULL | Завод-производитель / правообладатель торговой марки. Нужен для разруливания одноимённых брендов и для compliance. NULL допустим если ERP не различает бренд и производителя. |
| `weight_kg` | NUMERIC(10,3) | Вес единицы товара (с упаковкой). |
| `length_mm`, `width_mm`, `height_mm` | INTEGER NULL | Габариты единицы. **Без них не работает §4 ТЗ «Лимиты по объёму, паллетам, весу»** в калькуляции заказа. Если нет в ERP — soft, считаем объём как 0 (заказ не лимитируется по объёму, фиксируем риск). |
| `pallet_qty` | INTEGER NULL | Стандартное число consumer-единиц на паллете. На уровне карточки, потому что для большинства товаров укладка одинакова. Если бизнес подтвердит наличие нескольких паллетных конфигураций на один товар — выносим в `product_barcodes` (паллетный ITF-14). |
| `shelf_life_days` | INTEGER NULL | Заявленный срок годности **карточки** в днях (атрибут товара, не партии). Используется в калькуляции заказа (не везти больше, чем продастся до истечения), для распознавания списаний по сроку и для сверки с `expiry_date` партии в DESADV. Источник — поле «Срок годности» в ERP, нормализуем в дни (месяцы×30, годы×365). `NULL` = бессрочно (миска, поводок), не «неизвестно». Срок конкретной партии = `production_date + shelf_life_days` и хранится не здесь, а в движениях (`stock_movement`) при приёмке. **Из критики diagram_review §1.1.** |
| `storage_temp_min` / `storage_temp_max` | NUMERIC(4,1) | °C, **из критики §1.1**. Используется для проверки температурного режима на приёмке и для маршрутизации (рефрижератор vs обычный транспорт). |
| `requires_cold_chain` | BOOLEAN | Производное от `storage_temp_max < 8°C`, но материализуем для индексирования: «все товары холодовой цепи» — частый фильтр для логистики и QA-блокировок. |
| `requires_prescription` | BOOLEAN | Рецептурный ветпрепарат. Влияет на каналы продажи (не выкладывается в самообслуживание) и на формирование заказа в магазины без ветврача. Для зоомагазина это реальная категория. |
| `is_dangerous_goods` | BOOLEAN | Аэрозоли, бытовая химия, антипаразитарные жидкости — товары с ограничениями на транспортировку (ADR). Фильтр для маршрутизации поставок. |
| `uktzed` | TEXT | код УКТ ВЭД (для импорта/растаможки). |
| `category_path` | TEXT | Денормализованный путь от корня иерархии (см. 5.1.2) до листовой категории, разделители — ` / `. Хранится **как строка** для быстрого LIKE-фильтра и group-by без рекурсивных JOIN. Источник истины — таблица `category` (там parent_id), `category_path` пересчитывается при каждой выгрузке товара. Примеры: `Корма / Сухие / Для собак / Премиум`, `Аксессуары / Поводки и ошейники / Поводки`, `Ветаптека / Антипаразитарные / Капли на холку`. Длина до 512 символов; уровней обычно 3–5. Для фильтра «всё в категории Корма» используется `category_path LIKE 'Корма / %'`. |
| `status` | TEXT ENUM | **Глобальный** статус карточки в каталоге сети, бинарный: <br>• **`active`** — товар закупается хотя бы в одном магазине сети. Карточка участвует в выгрузке, прогнозе и калькуляции. <br>• **`archived`** — товар нигде не закупается и не возвращается в ассортимент (полное выведение из сети). Карточка остаётся в БД для исторических расчётов и retention, но `/v1/store_assortment` для неё пуст. <br>Жизненный цикл «вводится / распродаётся / заблокирован» **не здесь** — это per-store состояние и живёт в `store_assortment.lifecycle_state` (см. §5.1.4), потому что один и тот же товар в магазине №5 может распродаваться, а в магазине №12 оставаться активным. Переход `active → archived` допустим только когда все строки `store_assortment` для этого товара в `discontinued` (агрегатный инвариант, проверяется application-level + ночным контролем). |
| `created_at` | TIMESTAMPTZ NOT NULL DEFAULT now() | Когда строка впервые появилась в нашей БД (момент первой успешной загрузки этого `product_id`). **Immutable on UPSERT** — `ON CONFLICT DO UPDATE` его не трогает. Не путать с моментом создания карточки в ERP — это разные времена. |
| `updated_at` | TIMESTAMPTZ NOT NULL DEFAULT now() | Когда последний раз реально изменилось хоть одно tracked-поле. Обновляется UPSERT-ом только при фактическом diff (через `WHERE` clause или триггер `BEFORE UPDATE`), а не на каждой выгрузке. Нужен для индексации «что изменилось за последние N дней» без скана changelog. |
| `load_id` | UUID FK | Snapshot, в котором карточка пришла последний раз. **Обновляется на каждом UPSERT**, даже если бизнес-поля не менялись — это маркер «строка была видна в свежем экспорте, не залежавшаяся». Используется для алерта «карточка не приходила N дней» (кандидат на архивацию). |

##### 5.1.1.a Баркоды товара (`product_barcodes`)
Связь N×1: одна товар → ≥1 баркодов. Используется для lookup по чекам с касс и приёмкам.

Страна-производитель живёт **здесь**, а не в карточке товара: одна и та же позиция (тот же `product_id`) может выпускаться в нескольких странах, и каждая партия имеет **свой EAN** (первые 2–3 цифры GS1-префикса как раз кодируют страну регистратора). Поэтому пара «баркод → страна» однозначна, а «товар → страна» — нет.

| Поле | Тип | Notes |
|---|---|---|
| `barcode` | TEXT PK | EAN-13 / EAN-8 / UPC / внутренний |
| `product_id` | TEXT FK | → `products.product_id` |
| `barcode_type` | TEXT ENUM | `ean13` / `ean8` / `upc` / `itf14` / `internal` |
| `is_primary` | BOOLEAN | основной баркод (отображается на ценнике, обычно соответствует consumer-единице с `pack_qty = 1`) |
| `country_origin` | TEXT | ISO-3166 alpha-2, страна-производитель этой партии/SKU-варианта |
| `pack_qty` | INTEGER NOT NULL DEFAULT 1 | **Множитель к consumer-единице.** `1` — баркод самой единицы продажи (банка, пачка, поводок). `>1` — баркод групповой упаковки: `24` — блок из 24 банок, `144` — коробка/паллет от поставщика. При продаже на кассе scan баркода с `pack_qty=N` означает продажу N consumer-единиц одной операцией. Источник истины о фасовке — здесь, не в `products`. |
| `created_at` | TIMESTAMPTZ NOT NULL DEFAULT now() | Когда баркод впервые попал в нашу БД. Immutable on UPSERT. Полезно для разбора кейсов «откуда взялся этот EAN» и для отчётов по новым баркодам за период. |
| `updated_at` | TIMESTAMPTZ NOT NULL DEFAULT now() | Обновляется только при фактическом diff (смена `pack_qty`, `is_primary`, `country_origin`). |
| `load_id` | UUID FK | Snapshot последнего seen, обновляется всегда. Если баркод уезжает на другой `product_id` — это **critical** в severity-движке (вероятный merge карточек / ошибка маппинга, требует ручного разбора). |

Уникальность: `barcode` глобально уникален в рамках snapshot. Если ERP вернула дубль (один EAN на двух товарх) — **critical** в severity-движке.

#### 5.1.4. Ассортимент магазина (`store_assortment`) — пары товар × location
**Это ассортимент именно конкретного магазина** (или склада/даркстора — любой `location` нашей сети): какие товары числятся в данной торговой точке, плюс жизненный цикл этой связки. Один и тот же товар (`product_id`) обычно фигурирует во многих магазинах, но **в каждом — со своим состоянием** (где-то активен, где-то распродаётся, где-то ещё не введён, а в каких-то магазинах его нет вовсе). Это **не глобальный каталог сети** (для этого есть `products`) и **не остатки в штуках** (для этого есть `stock_snapshot` 5.2.2) — здесь только matrix-факт «товар × магазин» + lifecycle.

Отсутствие строки `(location_id, product_id)` означает **«никогда не было в этом магазине / never-stocked»** — это не статус, а отсутствие записи. Калькулятор такие пары не видит и не пытается заказать туда этот товар.

| Поле | Тип | Notes |
|---|---|---|
| `location_id` | FK → `location.location_id` | Магазин/склад/даркстор, в чьём ассортименте лежит товар. Пример: `STORE-005`. |
| `product_id` | FK → `products.product_id` | Карточка товара. Пример: `RC-ADULT-3KG` (Royal Canin для собак, 3 кг). |
| `min_shelf_presentation` | INTEGER | **MSP — минимальное товарное представление на полке.** Сколько штук должно быть видно покупателю в любой момент. Если фактический остаток падает ниже MSP — мерчандайзер докладывает, калькулятор повышает приоритет дозаказа. Пример: `MSP=4` для популярного корма (4 банки на полке всегда), `MSP=1` для дорогого редкого товара. |
| `facings` | INTEGER | **Фейсы — количество лицевых сторон товара на полке.** Влияет на скорость распродажи (визуальный объём) и на физическую вместимость. Пример: `facings=3` — товар занимает 3 «фасада» на стеллаже (3 в ряд × N в глубину). Для премиум-кормов часто `facings=1`, для дешёвых — `facings=4-6`. |
| `extra_display` | BOOLEAN | **Дополнительная выкладка** вне основной полки: торцевые стенды, прикассовая зона, паллет в проходе, сезонная выкладка. `true` — товар есть и в основной выкладке, и где-то ещё. Влияет на прогноз: extra-display даёт всплеск продаж +20–40%. Пример: на Новый год корма для собак ставят на торцевой стенд → `extra_display=true`. |
| `lifecycle_state` | TEXT ENUM NOT NULL DEFAULT `active` | **Per-store жизненный цикл пары.** Одно из: `pending` / `active` / `phasing_out` / `discontinued` (см. ниже). Пример: товар активен в магазине №5 (`active`), распродаётся в магазине №12 (`phasing_out`), не введён в магазин №20 (`pending`), выведен из №3 (`discontinued`). |
| `state_reason` | TEXT NULL | **Свободная причина последнего перехода lifecycle.** Заполняет категорийный менеджер или классификатор. Пример: `«пилот, тест на 4 магазинах»`, `«delisting под акцию Petshop»`, `«QA hold лот №2401, отзыв партии»`, `«сезон закрыт»`, `«ребрендинг — заменён на RC-ADULT-3KG-V2»`. |
| `introduced_at` | TIMESTAMPTZ NULL | **Дата ввода товара в этот магазин** (первая фактическая поставка или решение категорийщика). NULL пока `lifecycle_state='pending'` и физически ещё не приходило. Пример: `2026-03-15 00:00:00+00`. Используется для расчёта «новизны» (товар «new» = `now() - introduced_at < 28 дней`). |
| `discontinued_at` | TIMESTAMPTZ NULL | **Дата вывода из этого магазина.** NULL пока товар активен или распродаётся. Заполняется при переходе в `discontinued`. Пример: `2026-04-30 00:00:00+00`. После этой даты в `/v1/store_assortment?state=active` пара не возвращается. |
| `state_changed_at` | TIMESTAMPTZ NOT NULL | **Время последней смены `lifecycle_state`** (любой переход). Денормализация для быстрых запросов «что изменилось за последние N дней» без JOIN в журнал событий. Пример: `2026-05-01 14:23:00+00`. Если `lifecycle_state='active'` и `state_changed_at` > now() − 28d → товар считается «new». |
| `target_qty` | NUMERIC(10,3) NULL | **Целевой остаток на витрине + полке** (бизнес-цель, не факт). Калькулятор стремится заказать столько, чтобы `qty_on_hand + qty_in_transit ≈ target_qty + safety_stock`. `target_qty=0` сигналит run-out (используется классификатором для перевода в `phasing_out`). Пример: `target_qty=8` — держать 8 банок корма; `target_qty=0` — не закупать новые, распродать остаток. |
| `created_at` | TIMESTAMPTZ NOT NULL DEFAULT now() | Когда строка впервые появилась в нашей БД. Immutable on UPSERT. Пример: `2026-03-10 03:15:42+00` — это был день первого экспорта пары `(STORE-005, RC-ADULT-3KG)` из ERP. |
| `updated_at` | TIMESTAMPTZ NOT NULL DEFAULT now() | Когда последний раз реально изменилось бизнес-поле (MSP, facings, lifecycle_state, target_qty…). Не обновляется на «пустых» суточных загрузках без diff. |
| `load_id` | UUID FK → `loads.id` | Snapshot последнего seen. Обновляется на каждом UPSERT даже без diff (маркер «строка ещё видна в источнике»). Если `load_id` отстаёт от текущего на N дней — кандидат на архивацию. |
| **PK** | `(location_id, product_id)` | Одна строка на пару товар-магазин. Уникальность: один товар не может быть в одном магазине дважды. |

**Допустимые значения `lifecycle_state`:**
- **`pending`** — карточка зарегистрирована для магазина, но физически ещё не введена (нет первой поставки). Калькулятор: заказ по плану листинга, не по истории; safety stock завышен.
- **`active`** — нормальный режим, прогноз по истории продаж этой пары.
- **`phasing_out`** — товар распродаётся в этом магазине, новые заказы не формируются (`target_qty := 0`), маркдаун разрешён, lost-sales не считаем.
- **`discontinued`** — выведен из этого магазина, остаток выбыл (списан/вернули на РЦ/дораспродан). Строка остаётся для истории, но в выгрузке `/v1/store_assortment?state=active` не возвращается.

**Допустимые переходы** (CHECK + триггер):
```
pending      → active | discontinued
active       → phasing_out | discontinued
phasing_out  → discontinued | active   (повторное решение оставить)
discontinued → active                   (повторный ввод после паузы)
```
Запрещено: `active → pending`, `discontinued → phasing_out`, любой → `pending` после первого ввода.

**Что НЕ хранится колонкой, а вычисляется:**
- `is_new` = `lifecycle_state IN ('pending','active') AND state_changed_at > now() - interval '28 days'`. Окно «28 дней» — конфиг калькулятора, версионируется отдельно от схемы.
- `never_stocked` = отсутствие строки `(location_id, product_id)`.
- `out_of_season` — выводится из календаря категорий + текущей даты, не статус.

##### 5.1.4.a Журнал переходов жизненного цикла (`store_assortment_lifecycle_events`)
Append-only лог каждой смены `lifecycle_state`. **Это не SCD-2** (ADR-003 не нарушается): нет восстановления состояния на произвольную дату, только история событий для аудита и отчётов. Если позже понадобится полноценная bi-temporal история — backfill из этого журнала.

| Поле | Тип | Notes |
|---|---|---|
| `event_id` | UUID PK | Уникальный идентификатор события. Пример: `9f2d4c1a-8e3b-4c5f-9a1d-7b6e8f2a3c4d`. Генерится на нашей стороне (`gen_random_uuid()`), не из ERP. |
| `location_id` | TEXT FK → `location.location_id` | Магазин/склад, в котором произошёл переход. Пример: `STORE-005`. |
| `product_id` | TEXT FK → `products.product_id` | Карточка товара. Пример: `RC-ADULT-3KG`. Вместе с `location_id` ссылается на конкретную строку `store_assortment`. |
| `from_state` | TEXT NULL | Предыдущее значение `lifecycle_state`. **NULL только для первой записи** этой пары (когда товар впервые появился в магазине). Пример: `active`, `phasing_out`, `pending`. |
| `to_state` | TEXT NOT NULL | Новое значение `lifecycle_state`. Пример: `phasing_out` (переход active → phasing_out при решении вывести из магазина). |
| `changed_at` | TIMESTAMPTZ NOT NULL | Когда произошёл переход. Если изменение пришло из ERP — это время, указанное источником; если переход решили мы (auto_classifier) — `now()` на момент классификации. Пример: `2026-04-30 14:23:11+00`. |
| `changed_by` | TEXT | Кто инициировал переход. Допустимые форматы: <br>• `erp_sync` — пришло как факт из выгрузки ERP <br>• `category_manager:<user_id>` — явное решение категорийщика через UI / ручную правку в ERP. Пример: `category_manager:elena.k` <br>• `auto_classifier:<rule_id>` — наш классификатор по правилу из `lifecycle_rules.yaml`. Пример: `auto_classifier:rule_run_out_target_qty_zero` |
| `reason` | TEXT NULL | Бизнес-причина в свободной форме. Дублирует `store_assortment.state_reason` на момент перехода (для истории, если в текущей строке reason уже перезаписан). Примеры: `«delisting под Petshop»`, `«QA hold лот №2401»`, `«сезонная пауза до сентября»`, `«target_qty=0 в ERP 3 дня подряд»`. |
| `load_id` | UUID FK → `loads.id` | В каком прогоне выгрузки этот переход был зафиксирован. Пример: `8c1a4d2f-…`. Позволяет ответить «какие переходы попали в загрузку 2026-05-04 03:15». |

**Примеры строк:**

```
Сценарий: товар Royal Canin Adult 3kg в магазине STORE-005

(1) Ввод в магазин (первая запись):
  event_id = 9f2d4c1a-…
  location_id = STORE-005, product_id = RC-ADULT-3KG
  from_state = NULL,  to_state = pending
  changed_at = 2026-03-10 03:15:42+00
  changed_by = erp_sync
  reason = NULL

(2) Первая поставка пришла, перевели в active:
  from_state = pending,  to_state = active
  changed_at = 2026-03-15 09:00:00+00
  changed_by = auto_classifier:rule_first_receiving
  reason = «первая приёмка по DESADV-12345»

(3) Категорийщик решил вывести (под акцию конкурента):
  from_state = active,  to_state = phasing_out
  changed_at = 2026-04-30 14:23:11+00
  changed_by = category_manager:elena.k
  reason = «delisting под Petshop, перевод на собственную марку»

(4) Остаток дораспродан, выведен:
  from_state = phasing_out,  to_state = discontinued
  changed_at = 2026-06-01 22:00:00+00
  changed_by = auto_classifier:rule_qty_on_hand_zero
  reason = «qty_on_hand=0 7 дней подряд»

(5) Через год категорийщик вернул товар:
  from_state = discontinued,  to_state = pending
  changed_at = 2027-03-01 11:00:00+00
  changed_by = category_manager:elena.k
  reason = «возврат после переговоров с поставщиком»
```

Из такого журнала можно ответить на вопросы вроде:
- «Сколько SKU перешли в `phasing_out` за апрель?» → `WHERE to_state='phasing_out' AND changed_at BETWEEN ...`
- «Кто решил вывести этот товар?» → `WHERE location_id=... AND product_id=... AND to_state='discontinued'`
- «Какие переходы сделал авто-классификатор за вчера?» → `WHERE changed_by LIKE 'auto_classifier:%' AND changed_at >= ...`

## 7. Architecture & ADRs

### ADR-001 — Адаптер на нашей стороне (pull)
**Решение:** реализуем pull-адаптер на Go, который ходит в REST API ERP E-Zoo. E-Zoo не пишет outbound-логику.
**Trade-off:** контроль на нашей стороне vs нагрузка на ERP API. Принято: меньше зависимостей от E-Zoo dev-команды.

### ADR-002 — Materialize в PG18 (НЕ stream-through)
**Решение:** материализуем канонические таблицы в свой PG18.
**Trade-off:** хранение vs воспроизводимость. Принято: полнота, audit, восстанавливаемость > экономия на диске.

### ADR-003 — Bi-temporal только для фактов
**Решение:** факты (`receipt_line`, `stock_snapshot`, `stock_movement`) хранятся с `valid_time` + `system_time`. Мастер-данные — только current state (SCD-2 откладываем).
**Trade-off:** в первой итерации потеряем историю переименований товара, переноса баркодов между карточками и слияний категорий, но избежим x10 объёма master-данных.

### ADR-004 — REST-контракт с content-negotiation
**Решение:**
- Малые ответы (мастер-данные, `< 50 MB`) → `application/x-ndjson` inline + курсорная пагинация (`Link: rel=next`).
- Большие (`receipts`, `stock_movement`) → async-job: `POST /v1/exports {entity, snapshot_id}` → polling `GET /v1/exports/{id}` → 303 redirect на signed URL Parquet в S3.
**Trade-off:** один endpoint vs два кода. Принято: единый OpenAPI с двумя режимами по `Accept`.

### ADR-005 — URL-версионирование `/v1/...` + строгий OpenAPI
**Решение:** breaking change → `/v2`, additive → бэкпорт в `/v1`. OpenAPI лежит в репо, генерируем клиент для X-Flow.

### ADR-006 — Severity-движок валидации, config-driven
**Решение:** правила в `validation_rules.yaml`, severity = critical|soft. Стартовая политика — см. §4.3.
**Trade-off:** жёсткий старт (отрицательные остатки = critical) приведёт к фейлам в первый месяц, но это лучше чем тихая порча витрин.

### ADR-006a — Lifecycle классифицируется на нашей стороне, не тянется как готовый ярлык
**Решение:** из ERP тянем сырьё (`store_assortment.target_qty`, даты листинга, ссылки на промо-механику, глобальный «архив» карточки). Ярлыки `pending / active / phasing_out / discontinued` присваиваются нашим классификатором по правилам в `lifecycle_rules.yaml`. Версия правил пишется в `store_assortment.load_id` для воспроизводимости.
**Trade-off:** дополнительный код vs независимость от того, как именно конкретный товаровед E-Zoo заполняет поле статуса. Расцепляет наши релизы с ERP. Зависит от ответа на OQ-7 (если ERP даёт явный статус — добавляем `source='erp'` режим как fallback).

### ADR-007 — Atomic restart per load_id, 2 snapshot минимум
**Решение:** при сбое — откат всего `load_id`. X-Flow всегда видит committed snapshot. Хранится минимум 2.
**Trade-off:** потенциально ре-чтение 10M строк из ERP против resume-checkpoint. Принято: проще, идемпотентно; resume — отложенная оптимизация.

### ADR-008 — PII-стратегия: hash + drop
**Решение:** `loyalty_id` → `SHA256(id || salt)`; phone/email — drop. Salt в Vault, ротация ≤ раз в год.

### ADR-009 — Retention двухуровневый
**Решение:** 30d hot PG + 365d cold S3 (Parquet, partitioned by date).
**Trade-off:** дороже код (двухуровневый sweeper), но дешевле PG-storage и быстрее backfill.

### ADR-010 — Триггер: внутренний cron + admin-API
**Решение:** Go-cron внутри процесса + `POST /admin/loads` для manual/retry. K8s CronJob НЕ используем — меньше внешних зависимостей.

## 8. API contract (черновик OpenAPI)

```
GET  /v1/healthz
GET  /v1/snapshots                              # список доступных snapshot
GET  /v1/snapshots/current                      # текущий committed snapshot id

GET  /v1/products?since=...&cursor=...                 # NDJSON inline; .status = active|archived
GET  /v1/product_barcodes?since=...&cursor=...
GET  /v1/category, /v1/location, /v1/supplier,
GET  /v1/store_assortment?state=active,phasing_out&location_id=... # фильтр по lifecycle_state
GET  /v1/store_assortment/lifecycle_events?since=...               # журнал переходов
GET  /v1/master_change_log?entity=...&since=...&field=...    # diff-журнал мастер-данных (см. §4.5.a)
GET  /v1/supplier_edi, /v1/supply_spec, /v1/promo,
GET  /v1/supply_plan, /v1/order_rule

POST /v1/exports          {entity, snapshot_id, format=parquet}   → {export_id}
GET  /v1/exports/{id}     → {status, download_url?, parts?}

# admin
POST /admin/loads         {force=true?}         # ручной запуск
POST /admin/loads/{id}/retry
GET  /admin/loads/{id}    # статус, errors, reject_log summary
GET  /admin/reject-log?load_id=...&entity=...
```

Все ответы включают:
- `X-Snapshot-Id` / `X-Load-Id`
- `ETag` (хеш от `snapshot_id + cursor`)
- `Cache-Control: private, max-age=86400` (snapshot immutable)

## 10. Verification (как проверить, что работает)

1. **Unit + integration (dockertest):** PG18 контейнер, прогоняем full load на синтетическом источнике (mock REST), проверяем `loads.status='committed'`, `snapshot_pointer` сдвинулся, `reject_log` пуст для clean-input.
2. **Golden contract test:** OpenAPI v1 контракт фиксируется в репо; тест ломается при breaking change без bump версии.
3. **Severity-тесты:** 4 critical-кейса (FK miss, NULL PK, negative qty, future event) → load failed; 3 soft-кейса (NULL attr, bad format, EAN checksum) → load committed + reject_log.
4. **Atomic restart:** прерываем mock-источник на 50% → load failed → snapshot_pointer НЕ двинулся → следующий retry с нуля → committed.
5. **Read-during-write:** запускаем load, параллельно `GET /v1/products` → отдаёт previous snapshot, не видит running load.
6. **Performance smoke:** 1M receipt_line за 5 минут на dev-окружении (пропорционально 12M/час production).
7. **End-to-end в staging:** реальный 1С-mock с >100K товаром и ≥1.5 баркода на карточку (тест мульти-баркод lookup), измеряем длительность, объём rejects, размер Parquet-файлов.
8. **Multi-barcode lookup:** в моке создаём товар с 3 EAN, чек ссылается на каждый — все 3 строки разрешаются в одну `product_id`. Дубликат EAN на двух карточках → critical, load failed.

## 1. Что мы хотим получать

Раз в сутки (для начала — `batch`, ночное окно) — полный набор справочников и движений по всей сети. Это нужно прогнозу и калькулятору заказа. KPI, которые зависят от полноты и свежести: **OSA, OTIF, Stock days, доля автозаказов**.

### Состав данных (по группам)

**Мастер-данные:**
1. Товары (карточки): id, штрих-коды (несколько на товар), название, бренд, производитель, вес, габариты, упаковка, паллет, срок годности, температурный режим, УКТ ВЭД, иерархия категорий, статус (active/archived).
2. Иерархия категорий: id, parent_id, level, name.
3. Точки и склады: id, тип (магазин/склад/кросс-док/даркстор), GLN, адрес, GPS, график работы, ЕДРПОУ, приоритет по обороту.
4. Ассортимент магазина (пары товар × магазин): MSP, фейсы, доп. выкладка, целевой остаток, lifecycle-state (вводится / активен / распродаётся / выведен), даты ввода и вывода.
5. Поставщики: id, name, ЕДРПОУ/ІПН, GLN, payment terms.
6. Спецификации поставок (тройка поставщик × товар × локация): MOQ, закупочная цена, lead time, паллетная единица, приоритет поставщика.
7. Промо-акции: id, товар, тип (markdown / BOGO / bundle / loyalty / gift), даты, промо-цена с НДС.
8. Бизнес-правила заказов и план поставок (расписание поставок: какой поставщик в какой день недели возит куда).

**Факты (ежедневно):**
1. Чеки с касс (`receipt_line`): receipt_id, line_no, магазин, товар, сканированный баркод, qty, базовая цена, фактическая цена, сумма скидки, тип строки (sale/return/gift/promo_bonus), promo_id, time, hashed loyalty_id (без PII).
2. Остатки на конец дня (`location_stock_snapshot`): магазин × товар, qty_on_hand, qty_reserved, qty_in_transit.
3. Движения (`stock_movement`): приёмки (receiving) с привязкой к PO + lot_id + production_date + expiry_date + сравнение `qty_ordered/qty_shipped/qty_received/qty_damaged`; перемещения (transfer); списания (write_off) с reason_code; инвентаризации (inventory) с system_count vs physical_count; возвраты поставщику (return_to_vendor); брак при приёмке (damage).

Полная схема полей с описаниями и примерами — в [spec-2026-05-05.md §5](spec-2026-05-05.md).

## 2. Технические требования к контракту

Поддерживаем **один из** трёх вариантов (выбор за вами — что дешевле реализовать на стороне ERP):

### Вариант A — REST API (предпочтительный)
- Endpoint per entity, например `GET /api/products?since=...&cursor=...`.
- Курсорная пагинация (не offset/limit) — критично для крупных таблиц.
- ETag / If-Modified-Since для инкремента.
- JSON или JSONL (line-delimited JSON) для крупных коллекций.
- Сжатие gzip / br на ответе.
- HTTPS + одна из схем auth (см. §4).

### Вариант B — SQL read-only view + read-replica
- Доступ по сети к read-replica вашей ERP (PG / MSSQL / Oracle).
- Read-only пользователь с правами SELECT на views, которые вы соберёте под нашу схему.
- Одна view на сущность, плюс `_changes` view (или поле `updated_at`) для инкремента.
- Хорош, если ERP self-hosted и сеть безопасна (VPN site-to-site).

### Вариант C — SFTP-файлы
- Вы выкладываете в SFTP-папку файлы по расписанию (CSV / Parquet / JSONL).
- Манифест-файл в конце выгрузки с checksum'ами.
- Атомарность через rename `.tmp → .ok`.
- Худший по нагрузке (нет инкремента), но реалистичный для legacy-1С.

## 4. Аутентификация и сеть

Поддерживаем (выберите один по требованиям ИБ):
- **OAuth2 client_credentials** + JWT (предпочтительно).
- **mTLS** (взаимные сертификаты).
- **API-key в header + IP allowlist + TLS** (минимум).

Сеть — **VPN site-to-site** между вашей ERP и нашим хостингом (X-Flow cloud).

## 5. Гарантии данных, которые от вас просим

1. **Стабильные идентификаторы.** `product_id`, `location_id`, `supplier_id` не должны меняться при ребрендинге, переносе, переименовании. Если меняются — сообщайте через таблицу маппингов.
2. **Идемпотентность.** Повторный запрос той же выгрузки = тот же результат (если данные в источнике не менялись).
3. **Согласованность снимка.** Чеки одного дня + остатки на конец того же дня = согласованный snapshot (без race condition между ними).
4. **Без PII в чеках.** Номера карт лояльности — либо вообще не передавайте, либо в виде хеша. Телефоны / email — НЕ передавайте.
5. **Обозначайте удаления.** Если товар или магазин удалили из ERP — присылайте флаг `archived/deleted`, не просто пропуск из выгрузки (иначе мы не отличим «удалили» от «забыли передать»).

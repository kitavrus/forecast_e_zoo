Смотри, что я бы хотел. Чтобы на основании этих  документов, сделать минимальный MVP.
Поры будут состоять из нескольких модулей
1 - "Адаптер источников" - Будет состоять из двух частей.
1.1 - это джоба, которая будет ходить в ERP клиента 1 раз в сутки и забирать оттуда данные и сохранять их в (@docs/features/data-export/spec-2026-05-05.md) В таблице, которая тут описана. Важное замечание, что это модуль адаптер источников. То есть мы заранее должны заложить возможность изменения источника получения данных. То есть должны быть ДТОшки, должны быть мапперы. Это должно все учитываться даже на вот этой ранней стадии MVP.
1.2 - Вторая часть, которая описана (@docs/features/data-export/spec-2026-05-05.md) Это API, к которому подключаются другие модули и получают уже структурированную информацию, необходимую для дальнейших расчетов.

2 - "ETL и валидация" - В этом модуле мы получаем информацию из первого модуля. Делаем валидацию этих данных и сохраняем в новые таблицы для дальнейшей работы. У нас, грубо говоря, образуется второй слой. Уже подготовленных данных.
3 - "Витрины данных" - В этом модуле будет Joba, которая будет ходить во второй модуль и получать оттуда данные и строить витрину данных. Делаем ее максимально примитивной, плоской из нескольких таблиц. чтобы в дальнейшем при необходимости мы могли заменить с постгрыса, например, на кликхаус или на какой-то другой тип, или в паркете хранить. Для дальнейшей замены
4 - "KPI и калибровка OSA, OTIF, Stock days" - Этот модуль уже работает с витриной, в нем заданы в отдельных таблицах необходимые калибровки, необходимые KPI. И он этим занимается.
5 - "Прогноз → калькуляция → конструктор" - Этот модуль, учитывая результаты работы четвертого модуля и данных с 3 и 4, уже занимается непосредственно прогнозируем, калькуляцией и подготовкой данных. Для прогноза.
6 - "Блок подготовки заказа" - Этот модуль формирует заказы, подготавливает данные, тоже в своих это все делается в таблицах. И дальше подготовленные данные передаются в следующий модуль
7 - "Маршрутизация каналов EDI / 1С / ERP / CRM по описи поставщика" - Этот модуль у нас как и первый. (Он, как и первый, будет состоять из двух частей. Первая получает заказы, сформированные из нашей системы. Вторая часть отправляет IT заказы по API в систему клиента. ) Это уже адаптер заказов в зависимости от Того, куда мы должны их отправить у нашего клиента. То есть возможно будет только ERP. Сейчас мы делаем самый простой вариант, что мы просто по API подключаемся к ERP-системе нашего потенциального клиента и передаем ему заказы. В дальнейшем система должна быть адаптирована, так как это адаптер, чтобы. мог работать с разными системами клиента, которые здесь перечислены как пример. Но пока мы реализовываем только отправку. по API в систему клиентов
